```

│   ├── requirements.txt       ← Python-зависимости
│   ├── .env                   ← Переменные окружения (не в git)
│   └── .env.example           ← Шаблон для .env
│
├── frontend/
│   └── src/
│       └── pages/
│           └── Settings.jsx   ← Страница настроек с блоком лицензирования
│
├── nginx/
│   └── conf.d/default.conf    ← Конфигурация Nginx (реверс-прокси)
│
├── docker/
│   ├── entrypoint.sh          ← Скрипт запуска Docker-контейнера
│   └── supervisord.conf       ← Управление процессами внутри контейнера
│
├── tools/                     ← ⚠️ ТОЛЬКО ДЛЯ ВЕНДОРА. Клиенту не передавать
│   ├── gen_license.py         ← Генератор лицензионных ключей
│   └── build_commercial.sh    ← Скрипт сборки Docker-образа для клиента
│
├── scripts/
│   ├── backup.sh              ← Резервное копирование MongoDB
│   └── restore.sh             ← Восстановление из резервной копии
│
├── Dockerfile                 ← Описание сборки Docker-образа (3 стадии)
├── docker-compose.yml         ← ✅ Отдаётся клиенту. Запускает MongoDB + приложение

├── update.sh                  ← Скрипт обновления (bare metal)
└── manual.md                  ← Этот файл
```

### Ветки Git

| Ветка | Назначение |
|-------|-----------|
| `main` | Открытая версия без лицензирования. Для публичного репозитория |
| `commerce` | Коммерческая версия с полной системой лицензирования |

> **Правило:** все разработки ведёте в `commerce`. В `main` переносите только то, что хотите сделать публичным.

---

### Ключевые файлы подробно

#### `backend/server.py` — сердце системы
Содержит всю бизнес-логику, включая систему лицензирования:

| Функция/класс | Что делает |
|--------------|-----------|
| `get_machine_id()` | Вычисляет уникальный ID сервера: SHA256(`/etc/machine-id` + MAC-адрес), первые 32 символа |
| `compute_install_hash()` | HMAC-SHA256 от даты установки. Защита от ручного изменения даты в БД |
| `_validate_license_key()` | Проверяет лицензионный ключ по формуле: SHA256(`machine_id|expire_date|LICENSE_SALT`) |
| `verify_license()` | FastAPI-зависимость. Выбрасывает HTTP 402, если лицензия отсутствует или истекла |
| `GET /api/settings/license` | Возвращает статус лицензии: trial/active/expired, дни до истечения, Server ID |
| `POST /api/settings/activate` | Принимает лицензионный ключ, проверяет его и сохраняет |

`verify_license` подключена как зависимость ко всем POST/PUT/DELETE-эндпоинтам:
рискам, инцидентам, активам, угрозам, уязвимостям.

> Это означает: **просматривать** данные можно всегда, **изменять** — только при активной лицензии.

#### `tools/gen_license.py` — генератор ключей (только у вас)
Принимает Server ID клиента и дату истечения, возвращает лицензионный ключ.
Формат ключа: `YYYY-MM-DD.<sha256_hex_64_символа>`

#### `tools/build_commercial.sh` — скрипт сборки (только у вас)
Собирает Docker-образ с компиляцией `server.py` через Nuitka в `.so`-файл.
Исходный код Python в образ **не попадает**.

#### `docker-compose.yml` — отдаётся клиенту
Запускает два контейнера: `mongodb` и `securisk-app`. Клиент просто делает `docker compose up -d`.

#### `install.sh` — установщик для bare metal
Устанавливает SecuRisk на чистый Ubuntu-сервер без Docker.
Генерирует Machine ID и инициализирует лицензионную запись в MongoDB.

## 2. Как устроена система лицензирования

### Принцип работы

```
┌─────────────────────────────────────────────────────────────────┐
│                         КАК ЭТО РАБОТАЕТ                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  1. При установке:                                                │
│     install_date = текущая дата                                   │
│     machine_id   = SHA256(/etc/machine-id + MAC)[:32]            │
│     install_hash = HMAC(install_date, key=machine_id:SALT)       │
│     → всё записывается в MongoDB collection "settings"           │
│                                                                   │
│  2. При каждом изменении данных (POST/PUT/DELETE):               │
│     Бэкенд читает license_info из MongoDB                        │
│     │                                                             │
│     ├── нет license_key → проверяет дни с install_date          │
│     │   ├── ≤ 40 дней   → разрешает операцию                    │
│     │   └── > 40 дней   → HTTP 402 "Trial expired"              │
│     │                                                             │
│     └── есть license_key → проверяет:                           │
│         ├── SHA256(machine_id|expire|SALT) == key_hash?          │
│         ├── expire_date > today?                                 │
│         └── HMAC(install_date) совпадает?   ← защита от правки  │
│                                                                   │
│  3. Генерация ключа (вендор):                                    │
│     key = f"{expire_date}.{SHA256(machine_id|expire|SALT)}"     │
│     Ключ привязан к конкретному серверу и дате истечения         │
└─────────────────────────────────────────────────────────────────┘
```

### Что защищено

| Угроза | Защита |
|--------|--------|
| Клиент меняет `install_date` в MongoDB | HMAC-подпись: без знания SALT пересчитать нельзя |
| Клиент использует чужой ключ | Ключ привязан к `machine_id` конкретного сервера |
| Клиент читает исходники в Docker-образе | Nuitka компилирует `.py` → `.so`, исходник не попадает в образ |
| Клиент угадывает ключ | SHA256-хеш 64 символа: перебор невозможен |
| Timing attack при сравнении хешей | `hmac.compare_digest()` вместо `==` |

## 3. Инструкция для вендора (вы)

### 3.1 Первоначальная настройка

**Один раз, перед первой продажей:**

#### Шаг 1. Придумайте и сохраните секретную соль

Соль — это случайная строка, известная только вам. Она используется при генерации ключей.
**После выдачи первого ключа клиенту соль нельзя менять** — иначе все выданные ключи перестанут работать.

```bash
# Сгенерировать случайную соль (рекомендуется)
openssl rand -hex 32
# Пример: a3f7b2e1c4d5... (64 символа)
```

Сохраните соль в надёжном месте (менеджер паролей, зашифрованный файл).
Назовём её `МОЯ_СЕКРЕТНАЯ_СОЛЬ` в примерах ниже.

#### Шаг 2. Установите Python-зависимости для инструментов вендора

```bash
cd /opt/Projects/securisk
pip install hashlib  # уже встроен в Python, дополнительно ничего не нужно
```

---

### 3.2 Как собрать коммерческий образ

Образ собирается один раз (или при каждом обновлении продукта) и отправляется клиенту.

#### Вариант A: Быстрая сборка (для тестирования, без компиляции Nuitka)

```bash
cd /opt/Projects/securisk
SR_LICENSE_SALT="МОЯ_СЕКРЕТНАЯ_СОЛЬ" USE_NUITKA=false ./tools/build_commercial.sh
```

Занимает ~2-5 минут. Исходный `server.py` попадёт в образ — **не для клиентов**.

#### Вариант B: Production-сборка (с компиляцией через Nuitka)

```bash
cd /opt/Projects/securisk
SR_LICENSE_SALT="МОЯ_СЕКРЕТНАЯ_СОЛЬ" ./tools/build_commercial.sh
```

Занимает **~15-30 минут**. Nuitka компилирует `server.py` в бинарный `.so`-файл.
Исходный код Python **не попадает** в финальный Docker-образ.

#### Что скрипт делает

1. Проверяет, что `SR_LICENSE_SALT` задана и длиннее 16 символов
2. Запускает `docker build` с тремя стадиями:
   - **frontend-builder** → `yarn build` → статика React
   - **nuitka-builder** → компилирует `server.py` в `server.cpython-312-*.so`
   - **final** → копирует только `.so` и статику, исходники не копируются
3. Сохраняет образ в архив `securisk-latest.tar.gz`

#### Что отправить клиенту

```
securisk-latest.tar.gz    ← Docker-образ (несколько сотен МБ)
docker-compose.yml        ← Файл для запуска
```

> ❌ **Никогда не отправляйте:** `tools/`, `backend/server.py`, значение вашей соли.

### 3.3 Как сгенерировать лицензионный ключ для клиента

#### Шаг 1. Получите от клиента его Server ID

Клиент смотрит его в интерфейсе:
`Настройки → блок "Лицензирование" → поле "Server ID"`

Server ID выглядит так: `a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4` (32 символа hex)

#### Шаг 2. Запустите генератор ключей

```bash
cd /opt/Projects/securisk

SR_LICENSE_SALT="МОЯ_СЕКРЕТНАЯ_СОЛЬ" python tools/gen_license.py \
    a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4 \
    2027-01-31
```

Вы увидите:

```
============================================================
  Server ID   : a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4
  Действует до: 2027-01-31
  Ключ        : 2027-01-31.9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08
============================================================
Передайте строку 'Ключ' клиенту для активации в настройках.
```

#### Шаг 3. Передайте ключ клиенту

Отправьте клиенту только строку из поля **"Ключ"**:
```
2027-01-31.9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08
```

Ключ работает **только на том сервере**, с которого получен Server ID.
На другом сервере этот ключ не сработает.

#### Сроки лицензии — рекомендации

| Тип лицензии | Дата истечения |
|-------------|---------------|
| Годовая | `$(date -d "+1 year" +%Y-%m-%d)` |
| Квартальная | `$(date -d "+3 months" +%Y-%m-%d)` |
| Бессрочная | Укажите `2099-12-31` |

## 4. Инструкция для клиента с Trial-версией

> **Триал:** 40 дней полного доступа. Просматривать данные можно и после — изменять нельзя.

### Требования к серверу

| Параметр | Минимум |
|---------|--------|
| ОС | Ubuntu 22.04 LTS или 24.04 LTS |
| RAM | 2 ГБ |
| Диск | 20 ГБ |
| CPU | 2 ядра |
| Доступ | Интернет (для установки пакетов), SSH root |

### Установка

#### Шаг 1. Склонируйте репозиторий

```bash
git clone https://github.com/ваш-аккаунт/securisk.git
cd securisk
```

#### Шаг 2. Запустите установщик

```bash
sudo bash install.sh
```

Скрипт запросит:
- Пароль для базы данных MongoDB
- Пароль администратора системы

Установка занимает ~10-15 минут. В конце вы увидите:

```
==========================================
✅ ГОТОВО! Адрес: http://192.168.1.100
Логин: admin / Пароль: ВашПароль
------------------------------------------
Server ID для активации лицензии: a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4
Пробный период: 40 дней. Для активации обратитесь к поставщику.
==========================================
```

#### Шаг 3. Войдите в систему

Откройте в браузере `http://IP-вашего-сервера`
Логин: `admin` / Пароль: тот, что вы указали при установке

### Что происходит при истечении триала

Через 40 дней при попытке **добавить или изменить** данные появится ошибка:

> `Пробный период истёк. Активируйте лицензию в разделе Настройки.`

**Просматривать** существующие данные, отчёты и дашборд можно без ограничений.

Для продолжения работы:
1. Запишите ваш **Server ID** из раздела Настройки
2. Обратитесь к поставщику для получения лицензионного ключа

## 5. Инструкция для клиента с коммерческой версией

### Требования к серверу

| Параметр | Минимум |
|---------|--------|
| ОС | Любой Linux с Docker (Ubuntu 22.04+, CentOS 8+, Debian 11+) |
| RAM | 2 ГБ |
| Диск | 30 ГБ |
| CPU | 2 ядра |
| Docker | 24.0+ |
| Docker Compose | v2.0+ |

### 5.1 Развёртывание через Docker

#### Шаг 1. Установите Docker и Docker Compose

```bash
# Ubuntu/Debian
curl -fsSL https://get.docker.com | bash
sudo usermod -aG docker $USER
newgrp docker
```

Проверьте:
```bash
docker --version       # Docker version 24.x.x
docker compose version # Docker Compose version v2.x.x
```

#### Шаг 2. Загрузите предоставленные файлы на сервер

Вы должны были получить от поставщика два файла:
- `securisk-latest.tar.gz` — образ приложения
- `docker-compose.yml` — конфигурация для запуска

Загрузите их в удобную папку, например `/opt/securisk/`:

```bash
mkdir -p /opt/securisk
cd /opt/securisk
# Скопируйте сюда securisk-latest.tar.gz и docker-compose.yml
# Например через scp с вашего компьютера:
# scp securisk-latest.tar.gz docker-compose.yml user@сервер:/opt/securisk/
```

#### Шаг 3. Загрузите образ в Docker

```bash
cd /opt/securisk
docker load < securisk-latest.tar.gz
```

Вы увидите: `Loaded image: securisk:latest`

#### Шаг 4. Создайте файл с переменными окружения

```bash
cp docker-compose.yml docker-compose.yml.bak
nano /opt/securisk/.env
```

Содержимое файла `.env`:

```env
# Пароль для базы данных (придумайте сложный)
MONGO_ROOT_PASSWORD=УкажитеСложныйПароль123!
MONGO_PASSWORD=УкажитеДругойПароль456!

# Секретный ключ для JWT-токенов (произвольная случайная строка)
SECRET_KEY=случайная-строка-не-менее-32-символов-абвгдеёжз

# LICENSE_SALT встроена в образ вендором — НЕ указывайте здесь
```

Сгенерировать случайный `SECRET_KEY`:
```bash
openssl rand -hex 32
```

#### Шаг 5. Запустите систему

```bash
cd /opt/securisk
docker compose up -d
```

Дождитесь запуска (30-60 секунд):
```bash
docker compose ps
```

Оба контейнера должны быть в статусе `running`:
```
NAME                STATUS
securisk-mongodb-1  running
securisk-app-1      running
```

#### Шаг 6. Первый вход

Откройте в браузере: `http://IP-вашего-сервера`

Логин по умолчанию: `admin`
Пароль по умолчанию: `admin123`

> ⚠️ **Обязательно** смените пароль администратора сразу после входа!
> `Настройки → Пользователи → admin → Изменить пароль`

### 5.2 Активация лицензии в интерфейсе

После первого запуска система работает в **Trial-режиме** (40 дней).
Для активации полной лицензии:

#### Шаг 1. Узнайте ваш Server ID

1. Войдите в SecuRisk как администратор
2. Перейдите в раздел **Настройки** (боковое меню)
3. В самом верху страницы найдите карточку **«Лицензирование»**
4. Скопируйте значение поля **Server ID** (нажмите кнопку «Копировать»)

Server ID выглядит так: `a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4`

#### Шаг 2. Отправьте Server ID поставщику

Напишите поставщику ваш Server ID и укажите желаемый срок лицензии.

#### Шаг 3. Получите лицензионный ключ

Поставщик пришлёт вам строку вида:
```
2027-01-31.9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08
```

#### Шаг 4. Введите ключ в системе

1. Перейдите в **Настройки**
2. В карточке **«Лицензирование»** найдите поле **«Активация лицензии»**
3. Вставьте полученный ключ целиком (без пробелов)
4. Нажмите кнопку **«Активировать»**

При успешной активации статус изменится на:

```
✅ Активна  Лицензия активна до 31.01.2027
```

#### Шаг 5. Проверьте активацию

Попробуйте создать любую запись (риск, актив, инцидент). Если ограничений нет — лицензия работает.

## 6. Рабочий процесс продажи лицензии (шаг за шагом)

```
КЛИЕНТ                          ВЕНДОР (ВЫ)
  │                                 │
  │  1. Разворачивает систему       │
  │     (docker compose up -d)      │
  │                                 │
  │  2. Работает 40 дней бесплатно  │
  │                                 │
  │  3. Истекает триал / хочет       │
  │     полную версию               │
  │                                 │
  │ ─── Отправляет Server ID ──────>│
  │                                 │  4. Запускает:
  │                                 │     python tools/gen_license.py \
  │                                 │       <SERVER_ID> <EXPIRE_DATE>
  │                                 │
  │ <── Получает лицензионный ключ ─│
  │                                 │
  │  5. Вводит ключ в Настройках    │
  │     → Нажимает "Активировать"   │
  │                                 │
  │  ✅ Лицензия активирована        │
```

### Как продлить лицензию

За 2-4 недели до истечения:
1. Клиент снова передаёт вам **тот же** Server ID (он не меняется)
2. Вы генерируете новый ключ с новой датой истечения
3. Клиент вводит новый ключ — он перезаписывает старый

> **Важно:** Server ID остаётся неизменным, пока клиент не переустановит систему или не сменит сервер.

## 7. Частые вопросы и проблемы

### Вендор

**Q: Можно ли изменить соль после выдачи ключей клиентам?**
A: Нет. Если сменить `LICENSE_SALT`, все выданные ключи перестанут работать. Храните соль бессрочно.

**Q: Как выдать ключ на новый сервер (клиент переехал)?**
A: Клиент устанавливает систему заново, получает новый Server ID, вы генерируете новый ключ. Старый ключ на новом сервере не работает.

**Q: Как выпустить обновление продукта?**
A: Измените код в ветке `commerce`, соберите новый образ тем же скриптом (с той же солью). Клиент делает:
```bash
docker load < securisk-новая-версия.tar.gz
docker compose pull  # или: docker compose up -d --force-recreate
```

**Q: Как проверить, что в образе нет исходников Python?**
```bash
docker run --rm securisk:latest find /app/backend -name "*.py" 2>/dev/null
# Ожидаемый результат: пусто (только .so файл)
```

### Клиент

**Q: Ошибка "Trial period expired" при попытке изменить данные**
A: Пробный период 40 дней истёк. Просматривать данные можно. Для продолжения — обратитесь к поставщику за лицензией.

**Q: Ошибка "Неверный лицензионный ключ"**
A: Проверьте:
- Ключ скопирован полностью (64 символа хеша + дата)
- Нет лишних пробелов в начале/конце
- Ключ получен именно для **этого** сервера (Server ID совпадает)

**Q: Сервер переехал на новое железо. Ключ перестал работать**
A: Да — ключ привязан к железу (Machine ID). Сообщите поставщику новый Server ID для перевыпуска ключа.

**Q: Как посмотреть логи приложения?**
```bash
docker compose logs -f app      # логи приложения
docker compose logs -f mongodb  # логи базы данных
```

**Q: Как создать резервную копию данных?**
```bash
docker compose exec mongodb mongodump \
    --db securisk_db \
    --out /tmp/backup_$(date +%Y%m%d)
docker compose cp mongodb:/tmp/backup_$(date +%Y%m%d) ./backup/
```

**Q: Как перезапустить систему?**
```bash
docker compose restart
# Или полный перезапуск:
docker compose down && docker compose up -d
```

**Q: Как обновить систему (когда поставщик пришлёт новый образ)?**
```bash
cd /opt/securisk
# Загрузите новый архив от поставщика в /opt/securisk/
docker load < securisk-новая-версия.tar.gz
docker compose up -d --force-recreate app
```
> ⚠️ Данные в MongoDB хранятся в Docker Volume и при обновлении **не теряются**.

## Приложение: Структура лицензионного ключа

```
2027-01-31.9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08
│          │
│          └─ SHA256(machine_id|expire_date|LICENSE_SALT) — 64 символа
│
└─ Дата истечения в формате YYYY-MM-DD
```

Проверка на стороне бэкенда:
1. Разбить ключ по первой точке → `expire_date` и `key_hash`
2. Пересчитать: `expected = SHA256(f"{machine_id}|{expire_date}|{LICENSE_SALT}")`
3. Сравнить `key_hash == expected` через `hmac.compare_digest()`
4. Проверить `expire_date > today()`

Ключ невозможно подобрать или подделать без знания `LICENSE_SALT`.
